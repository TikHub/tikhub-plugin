# TikHub Plugin for Claude Code — Overview & Architecture

> **This is the master design document.** It is the brainstorm + architecture + roadmap.
> The actual step-by-step, TDD-style implementation plans live in the sibling files
> `2.*.md` … `6.*.md` (see [Roadmap](#9-roadmap--plan-files)). Read this first, then execute
> the numbered plans in order.

**Goal:** Ship `tikhub-plugin` — an official Claude Code plugin that gives Claude (and users)
first-class access to TikHub's social-media data API: download media, fetch profiles/posts/
comments, search, and run trend/competitor/listening analyses across 16+ platforms — through
TikHub's **official hosted MCP server**, direct REST, or the Python SDK.

**Architecture:** A skills-centric plugin (like the Bright Data plugin) with three layers:
(1) a **foundation layer** that wires up the integration mechanisms and routes the user,
(2) a **platform-coverage layer** with deep skills for the six biggest platforms plus an
endpoint-discovery skill that unlocks the remaining ~50 API tags, and (3) a **task layer** of
high-value, outcome-oriented skills (downloader, creator analytics, trend research, social
listening, etc.). The plugin **does not ship its own MCP server** — it points `.mcp.json` at
TikHub's existing hosted MCP (`mcp.tikhub.io`); REST and the Python SDK are documented
alternatives.

**Tech stack:** Claude Code plugin format (`plugin.json`, `marketplace.json`, `SKILL.md`,
`.mcp.json`); TikHub REST API V5.3.2 (`https://api.tikhub.io`); TikHub's official hosted MCP
(`mcp.tikhub.io`, 16 per-platform servers, 924+ tools; source =
`~/PycharmProjects/TikHub_MCP`); TikHub Python SDK (`pip install tikhub`, v2.1.x); Bash helper
scripts + a bundled trimmed OpenAPI index for endpoint discovery.

## 1. Decisions locked (from brainstorming)

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Primary integration mechanism | **MCP-first, all three** — `.mcp.json` points at TikHub's **official hosted MCP** (`mcp.tikhub.io`, via `npx mcp-remote`); REST-direct and Python SDK documented | MCP gives Claude direct tool access to endpoints; REST is the zero-install fallback; SDK serves app builders. Mirrors the proven Bright Data pattern. |
| 1a | Build our own MCP? | **No.** Reuse TikHub's existing hosted MCP (16 per-platform servers, 924+ tools, covers all tags). Self-host (`pip install tikhub-mcp` / Docker) documented as an alternative | TikHub already ships a complete, production MCP (source: `~/PycharmProjects/TikHub_MCP`). Rewriting it would duplicate work and drift from the canonical tool set. |
| 2 | v1 platform depth | **Core 6 deep** (TikTok, Douyin, Instagram, YouTube, Twitter/X, Xiaohongshu) **+ endpoint-discovery** for the other 10+ | 1,107 endpoints across ~57 tags is too many for one-skill-per-tag. Deep where demand is highest; discovery so nothing is blocked. |
| 3 | Value emphasis | **Balanced**: task/use-case skills **and** platform/endpoint-coverage skills | Highest perceived value and matches Bright Data's mix of integration + outcome skills. |
| 4 | Plugin name / namespace | `tikhub-plugin` (skills invoked as `/tikhub-plugin:<skill>`) | Matches `brightdata-plugin` convention. |
| 5 | Distribution target | `claude-community` marketplace (submit via in-app/Console form); ship own `marketplace.json` too | Official marketplace is curated by Anthropic with no application; community is the open path. |

## 2. What TikHub is (facts that drive the design)

- **Scale:** OpenAPI spec **V5.3.2**, **1,107 endpoints** (948 GET / 159 POST) across **~57 tags**.
- **Platforms (16+):** TikTok, Douyin, Xiaohongshu (RedNote), Lemon8, Bilibili, Kuaishou,
  PiPiXia, Weibo, WeChat (Channels + Media Platform), Toutiao, Xigua, Instagram, YouTube,
  Twitter/X, Threads, Reddit, LinkedIn, Zhihu — plus **Temp-Mail** and a **Captcha solver**.
- **Top tags by endpoint count:** TikTok-App-V3 (75), Douyin-Web (68), TikTok-Web (61),
  Douyin-App-V3 (45), Douyin-Index (44), LinkedIn (43+42), TikTok-Ads (35), Zhihu (34),
  Weibo (33), Instagram (32+29+27), Bilibili (30), Reddit (28), Kuaishou (26),
  YouTube (25+21), Xiaohongshu (multiple), … (full table goes in the endpoint-discovery reference).
- **Capability shapes per platform:** single video/post fetch · user profile · post/like/
  collection/repost lists · followers/following · search (video/user/hashtag/live) · trending
  & billboards · comments + replies · home feed · ads · creator analytics · shop.
- **Auth:** `Authorization: Bearer <API_KEY>`. Keys created at https://user.tikhub.io
  (up to 20 keys/account). Email-verified registration required.
- **Base URLs:** `https://api.tikhub.io` (global) · `https://api.tikhub.dev` (Mainland China).
- **Limits:** QPS 10/sec; request timeout 30–60s; up to 3 automatic retries recommended.
- **Billing:** credit/balance system, **per-endpoint pricing**, daily-volume tier discounts,
  a pricing-calculator endpoint, free daily check-in credits, PayPal + USDT payment,
  referral credit. **Cost-awareness is a cross-cutting concern** (see §6).
- **Already-built plumbing we will lean on:**
  - **Python SDK** — `pip install tikhub`, v2.1.2, 100% endpoint coverage, namespaced client
    (`from tikhub import TikHub; client = TikHub(api_key=...); client.douyin_web.fetch_one_video(aweme_id=...)`),
    ~54 resources, async/sync, retries+backoff, structured errors.
  - **Official MCP (hosted) — we connect to this; we do NOT build one.** TikHub MCP Server v2
    is **16 platform-based MCP servers exposing 924+ endpoints**, split per platform to fit LLM
    context windows. Hosted at `https://mcp.tikhub.io/{platform}/mcp`.
    - **Connect (recommended, remote):** `npx mcp-remote https://mcp.tikhub.io/{platform}/mcp
      --header "Authorization: Bearer ${TIKHUB_API_KEY}"` (stdio bridge). SSE / Streamable-HTTP
      transports also available. "Add only the platforms you use."
    - **Self-host (alternative):** `pip install tikhub-mcp` then
      `tikhub-mcp --platform tiktok --transport stdio`, or run the all-platform HTTP hub
      (`--transport http --port 8000` → `http://localhost:8000/{platform}/mcp`), or Docker.
    - **Platform slugs (16):** `tiktok` (204 tools), `douyin` (247), `instagram` (56),
      `xiaohongshu` (50), `weibo` (64), `bilibili` (38), `kuaishou` (33), `zhihu` (32),
      `linkedin` (25), `reddit` (24), `youtube` (21), `wechat` (19), `others` (64 —
      Lemon8/PiPiXia/Xigua/Toutiao/Sora2), `tikhub` (23 — account/downloader/health/demo),
      `twitter` (13), `threads` (11).
    - **Env:** `TIKHUB_API_KEY` (required); optional `TIKHUB_BASE_URL`, cache + rate-limit vars.
    - **Source of truth:** `~/PycharmProjects/TikHub_MCP` (`src/platforms/*`, `cache/endpoints.json`).

## 3. Who uses it & primary use cases

**Personas**
- **Marketer / analyst** — "What's trending on TikTok this week?", "Monitor mentions of our brand."
- **Creator / agency** — "Analyze this creator's last 100 posts", "Benchmark us vs. 3 competitors."
- **Data engineer / app builder** — wiring TikHub into a pipeline via the SDK or REST.
- **Casual / power user** — "Download this video without watermark", "Export these comments."

**Flagship use cases (become task skills)**
1. Download media (video/audio/images, no-watermark) from any supported URL.
2. Analyze a creator/account across posts, engagement, and growth.
3. Discover trending/viral content and rising hashtags.
4. Social listening: brand/keyword mentions → sentiment → themes → cited digest.
5. Competitor benchmarking across accounts/platforms.
6. Hashtag/keyword research.
7. Comment mining & sentiment for a post.
8. Bulk export of posts/followers/search results to CSV/JSON.

## 4. Integration strategy (MCP-first, all three)

The `tikhub-onboarding` skill is the entry point and **router**. After a one-time auth setup it
sends the user down one of three paths:

| Path | When | Mechanism | Owning skill |
|------|------|-----------|--------------|
| **MCP (default)** | Claude should call endpoints directly as tools | `.mcp.json` connects to the **hosted** `mcp.tikhub.io/{platform}/mcp` via `npx mcp-remote`; `${TIKHUB_API_KEY}` from env | `tikhub-mcp` |
| **REST-direct** | Zero-install, scripting, CI, non-Python | `curl` with `Authorization: Bearer` against `api.tikhub.io` | `tikhub-rest-api` |
| **Python SDK** | Building an app/pipeline | `pip install tikhub`, namespaced client | `tikhub-python-sdk` |

**Bundled MCP config (`.mcp.json` at plugin root)** — points at TikHub's hosted MCP, one entry
per platform (ship the Core-6 enabled; document how to add the other 10). No server to build or
package — just the remote bridge:
```json
{
  "mcpServers": {
    "tikhub-tiktok": {
      "command": "npx",
      "args": ["-y", "mcp-remote", "https://mcp.tikhub.io/tiktok/mcp",
               "--header", "Authorization: Bearer ${TIKHUB_API_KEY}"]
    },
    "tikhub-douyin": {
      "command": "npx",
      "args": ["-y", "mcp-remote", "https://mcp.tikhub.io/douyin/mcp",
               "--header", "Authorization: Bearer ${TIKHUB_API_KEY}"]
    }
    // … instagram, youtube, twitter, threads, xiaohongshu (Core-6); others documented
  }
}
```
> **Resolved (was the old #1 risk):** TikHub's MCP is a **hosted remote** service, not an
> OS-specific binary, so it bridges cleanly with `npx mcp-remote` and needs no bundling. The
> only Plan-2 confirmations: (a) exact `${TIKHUB_API_KEY}` substitution behavior in plugin
> `.mcp.json`, (b) whether to enable all 16 platform entries by default or just Core-6 +
> documented opt-in (default: Core-6, to keep the tool surface lean), and (c) document the
> `pip install tikhub-mcp` self-host fallback in `tikhub-mcp` for offline/air-gapped users.

## 5. Skill architecture

~19 skills in three layers (Bright Data ships 21, for reference).

### Layer A — Foundation (5)
| Skill | Responsibility |
|-------|----------------|
| `tikhub-onboarding` | Entry point: register/get API key, set `TIKHUB_API_KEY`, pick a path (MCP/REST/SDK), route to the right skill. Mirrors Bright Data `agent-onboarding`. |
| `tikhub-mcp` | Connect to & orchestrate TikHub's **official hosted MCP** (`mcp.tikhub.io`, 16 per-platform servers): how to add platform entries, the `mcp-remote` bridge, the `pip install tikhub-mcp`/Docker self-host fallback, tool catalog, troubleshooting. **Does not build an MCP.** |
| `tikhub-rest-api` | REST best practices: base URLs, Bearer auth, pagination (cursor/offset/`max_cursor`), QPS 10 + retry/backoff, error codes, **credit-cost awareness + pricing calculator**, China endpoint. |
| `tikhub-python-sdk` | SDK guide: install, client init, the ~54 namespaces, sync/async, error hierarchy, common patterns. |
| `tikhub-endpoint-discovery` | **Unlocks all 1,107 endpoints.** Bundles a trimmed OpenAPI index + a `bin/` search helper to find the right endpoint by platform/intent. The catch-all for the ~50 tags not covered by a dedicated platform skill. |

### Layer B — Platform coverage (Core 6)
Each skill maps that platform's tags to capability tables (fetch / profile / lists / search /
trending / comments) with example calls for all three paths and a `references/endpoints.md`.

> **Coverage scope (per user direction):** each platform skill documents only the API
> versions/tags below. The de-scoped tags (e.g. TikTok-Web, the older Instagram versions) are
> still reachable via the hosted MCP and the `tikhub-endpoint-discovery` skill — we just don't
> write dedicated reference material for them in v1.

| Skill | Tags covered (v1) | De-scoped (removed from this skill) |
|-------|-------------------|-------------------------------------|
| `tiktok` | TikTok-App-V3, TikTok-Ads, TikTok-Creator, TikTok-Analytics, TikTok-Shop-Web | TikTok-Web |
| `douyin` | Douyin-App-V3, Douyin-Index, Douyin-Search, Douyin-Billboard, Douyin-Xingtu (+V2) | Douyin-Web, Douyin-Creator (+V2) |
| `instagram` | Instagram-V2 | Instagram-V1, Instagram-V3 |
| `youtube` | YouTube-Web-V2 | YouTube-Web |
| `twitter-threads` | Twitter-Web, Threads-Web | — |
| `xiaohongshu` | Xiaohongshu-App-V2 | Xiaohongshu Web V1/V2/V3, App (V1) |

### Layer C — Task / use-case (8)
| Skill | Outcome |
|-------|---------|
| `social-media-downloader` | Download video/audio/images (no-watermark) from any supported URL. Uses Hybrid-Parsing + per-platform fetch. The headline feature. |
| `creator-analytics` | Profile + post history + engagement + growth for an account, across platforms. |
| `trend-research` | Trending feeds, billboards (Douyin Billboard/Xingtu), rising hashtags & viral content. |
| `social-listening` | Brand/keyword mentions across platforms → sentiment → theme clustering → cited digest. Mirrors Bright Data `brand-listening`. |
| `competitor-analysis` | Benchmark N accounts/competitors across platforms (followers, cadence, engagement). |
| `hashtag-research` | Hashtag/keyword search, related tags, volume signals, top content under a tag. |
| `comments-analysis` | Pull comments+replies for a post → sentiment, themes, top commenters. |
| `bulk-data-export` | Paginate large lists (posts/followers/search) with dedup → CSV/JSON for downstream analysis. |

## 6. Cross-cutting concerns (apply to every skill)

1. **Setup gate** — every skill checks `TIKHUB_API_KEY` is set (and MCP/SDK present for those
   paths) before acting; if missing, halt and route to `tikhub-onboarding`. (Pattern copied
   from Bright Data's per-skill "Setup gate".)
2. **Cost awareness** — endpoints are billed per call. Skills must (a) warn before large
   paginated/bulk pulls, (b) prefer a single batched endpoint over N single fetches when one
   exists, and (c) reference the pricing-calculator endpoint. `bulk-data-export` and
   `social-listening` carry explicit cost callouts.
3. **Verification gate** — never claim success without inspecting the response: non-empty,
   not an auth/credit error, expected fields present (e.g. a video fetch returns a play URL).
4. **Rate limiting** — respect QPS 10/sec; show retry/backoff for loops.
5. **Region** — surface the `api.tikhub.dev` base for Mainland-China users.

**Rules from the Claude plugin docs (do not violate):**
- Only `plugin.json` goes inside `.claude-plugin/`. Everything else (`skills/`, `.mcp.json`,
  `bin/`, `hooks/`) lives at the plugin **root**.
- Each skill is `skills/<name>/SKILL.md` with YAML frontmatter (`name`, `description`).
- Set an explicit `version` in `plugin.json` so users get controlled updates.
- Run `claude plugin validate` before any submission.

## 8. Testing & quality strategy

Skills are Markdown, so "tests" are structural + behavioral:
- **`scripts/validate-skills.sh`** (TDD anchor for Plan 2): asserts every `SKILL.md` has valid
  frontmatter with `name` + `description`; every `references/...` path mentioned in a SKILL
  exists; `.mcp.json`, `plugin.json`, `marketplace.json` are valid JSON; `openapi-index.json`
  parses and is non-empty.
- **`claude plugin validate`** — official validator, run in CI and before submit.
- **Local load test** — `claude --plugin-dir ./tikhub-plugin`, then exercise
  `/tikhub-plugin:tikhub-onboarding` and one platform + one task skill end-to-end against a
  live key (manual checklist in Plan 6).
- **Endpoint-index freshness** — a regenerate step (Plan 3) rebuilds `openapi-index.json` from
  the live `openapi.json` so the catalog stays current with the API version.

## 10. Open questions / risks

1. **MCP packaging — RESOLVED.** ~~Highest risk~~. TikHub's MCP is a **hosted remote** service
   (`mcp.tikhub.io/{platform}/mcp`), so `.mcp.json` bridges it with `npx mcp-remote` — no
   bundling, no binary, no fallback needed. Remaining minor checks (key substitution, default
   platform set, self-host docs) are in Plan 2. We do **not** build our own MCP.
2. **`npx mcp-remote` dependency:** the MCP path needs Node/`npx` available. Mitigation:
   document the `pip install tikhub-mcp` self-host path and the REST path for Node-less envs.
3. **Endpoint-index size:** the full `openapi.json` is ~3 MB. The trimmed index must stay small
   enough to ship in-repo (target: path+method+tag+summary+key params only). The MCP repo's
   `cache/endpoints.json` is a ready-made starting point.
4. **Credit cost in demos:** examples/tests consume real credits. Use the cheapest endpoints
   (e.g. single fetch) and the free daily check-in credits for the manual test pass.
5. **API drift:** spec is at V5.3.2 today; add the regenerate step so the index can be refreshed.
6. **Platform promotion:** confirm the Core-6 choice matches actual user demand before
   investing in deferred-platform skills.
